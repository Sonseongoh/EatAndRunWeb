# 웹/PWA로 완수 루프를 먼저 검증하고, 검증되면 네이티브로 간다

핵심 활동(나가서 걷고 뛰기)은 본질적으로 모바일이고, 이 제품의 두 약점 — 백그라운드 GPS 추적, 푸시 알림 — 은 정확히 네이티브 앱이 해결해주는 것들이다. 따라서 장기적으로 이 제품은 모바일 앱이 맞다. 그러나 "앱이 맞다"가 "지금 네이티브로 가야 한다"는 아니다.

아직 검증되지 않은 핵심 가설은 **"사람들이 실제로 완수(Completion)를 누르는가, 즉 다리가 닫히는가"**이다. 검증 안 된 가설에 네이티브 재작성이라는 큰 베팅을 걸지 않는다. 대신 **기존 Next.js 코드를 PWA로 만들어**(홈 화면 설치 + wake lock + 포그라운드 GPS + 안드로이드 웹푸시) 완수 루프를 싸게 검증한다. 완수율이 증명되면, 그때 네이티브의 GPS 자동 판정·신뢰성 있는 푸시는 *검증된 수요로 정당화되는 다음 플랫폼*이 된다.

이 결정은 ADR-0001(수동 완수 먼저)·ADR-0002(푸시 대신 재방문 직면)와 같은 원칙의 상위 노드다: 반쯤 부서진 카드에 핵심을 걸지 말고, 견고한 버전부터 출시해 검증한다.

## Considered Options

- **웹/PWA 먼저(채택)** — 기존 코드 활용, 저비용 검증. 백그라운드 GPS·iOS 푸시는 약하지만, 완수 루프 검증에는 불필요.
- **지금 네이티브 전환(거부)** — 정공법이지만 미검증 가설에 큰 베팅. 검증 후로 미룬다.
- **하이브리드(입구는 웹, 추적만 앱)(거부)** — 복잡도 과다.
