**소개** — 음식 분석을 입구로, "방금 먹은 것 → 지금 당장의 행동 → **실제 완수**"까지 잇는 실행형 건강 루틴. 측정에서 멈추는 칼로리 앱과 음식 맥락이 없는 러닝 앱 사이의 빈틈을 노렸다.

## ① 문제 정의 & 성공 지표
- **문제** — 사용자는 "얼마 먹었다"는 알아도 "그래서 지금 뭘 해야 하나 → 실제로 했나"로 이어지지 않는다. 측정과 행동 사이가 끊겨 있다.
- **관점** — 이 제품의 가치는 측정이 아니라 **다리(Bridge)**: 먹은 것을 지금 당장의 구체적 행동으로 바꿔 *실제로 하게* 만드는 것.
- **성공 지표** — 성공을 '기록을 남겼는가'가 아니라 **완수율(계획→실제 완수 전환율)**로 정의. 이 지표가 이후 모든 의사결정의 기준이 됐다.

## ② 구현 범위
- **음식 분석** — 사진 업로드·텍스트 입력 두 방식, 분석 상태 표시, 칼로리 범위·평균으로 가공해 다음 단계로 연결
- **운동 방식 선택** — 체중·페이스·소모비율에 따라 운동 시간/강도 실시간 반영
- **경로 추천** — Google Maps 연동, 2개 추천 경로 표시·선택, 운동 데이터와 함께 저장
- **운동 기록** — 분석·운동·경로를 한 기록으로 저장/조회, 필터·무한스크롤
- **완수 루프** *(심화)* — 1탭 완수 체크, 당일 미완수는 '놓침' 처리, 재방문 시 '오늘의 미완료' 직면, 연속 기록(streak)
- **완수율 지표** *(심화)* — 완수 ÷ (완수+놓침)으로 완수율 카드 제공 (오늘 미완료는 분모 제외)
- **PWA 설치형 셸** *(심화)* — 매니페스트·서비스워커·설치 배너로 홈 화면 설치 지원

## ③ 핵심 의사결정 *(선택지 → 결정 → 근거)*
**1. 완수 판정 — GPS 자동 → 수동 1탭 먼저**
웹은 백그라운드 위치 추적이 불가(화면 종료 시 JS 중단)해 자동에 베팅하면 완수 고리가 영영 안 닫힌다. **검증 안 된 가설에 비싼 기능을 먼저 걸지 않고**, 수동 1탭으로 닫고 GPS 자동은 향상 레이어로 분리.

**2. 재참여 — 푸시 → 인앱 '오늘의 미완료' 직면 + 연속기록**
웹·익명 환경에서 푸시는 신뢰할 수 없는 채널(iOS 제약 등). "사용자를 능동적으로 부를 수 없다"를 인정하고, 돌아왔을 때 완수를 *쉽고 닫고 싶게* 만드는 쪽으로 설계.

**3. 정체성 — 로그인 강제 → 익명 우선 + 전환 훅**
분석은 비용(OpenAI 호출)이라 루프는 로그인 기능으로 두되, **익명으로 입구 마찰을 0으로** 유지하고 streak이 쌓이면 "저장하려면 로그인"으로 전환. 입구의 즉시성과 후크의 견고함을 동시에.

**4. 칼로리·운동량 추정 — "정확"이 아니라 "정직한 추정"**
ACSM 표준 MET 공식으로 계산. 핵심 통찰: **사전 추정은 사용자의 실제 속도를 알 수 없어 본질적으로 추정치**다(페이스를 입력받아도 본인이 모름 → GIGO). 그래서 모드별 고정 MET가 오히려 정직한 기본값. 그 위에서 *칼로리(MET)와 경로 시간(페이스)이 서로 다른 속도를 가정하던 내부 불일치*를 발견해 8.4km/h로 통일했다. 그리고 "**진짜 정확도는 사후 GPS 측정에서 온다**"는 결론이 GPS/완수 측정 방향을 정당화한다 — 사전 계산을 더 정교화하는 대신 측정으로 가는 게 맞다.

**5. 경로 추천 — 왕복 기준 + "다양성"으로서의 다중 코스**
편도 추천은 사용자를 목적지에 *버려두고* 목표 칼로리와도 어긋나, **왕복(out-and-back)**으로 통일(표시값·길 안내 전부). 그리고 다중 코스의 가치는 거리·방위 차이(가짜 차별점)가 아니라 **"원하는 길을 고르는 다양성"**이라 판단 → 요청마다 *무작위 방향* 2개를 충분히 벌려 뽑고(도로 없는 방향은 재시도), 선택은 지도로. 거리 정확도는 직선→도로 보정 휴리스틱으로 근사하고, 정밀 보정(API 2배)은 비용 때문에 보류.

## ④ 기술적 깊이 / 트러블슈팅
- **도메인 로직 순수 함수화 + 테스트 도입** — 완수·놓침·연속기록 판정을 KST 날짜 경계 기준 순수 함수로 분리하고 **Vitest 도입**, 자정 경계·연속기록 유예·하루 다중 계획 등 엣지케이스를 단위 테스트로 고정.
- **서비스워커 미등록 버그** — PWA 설치가 안 되는 원인 추적: 등록 코드가 `window load`를 기다렸으나 하이드레이션 시점엔 이미 발생 완료라 콜백 미호출 → 즉시 등록으로 수정.
- **죽은 코드 발견** — 게스트 전환 화면이 `/login` 리다이렉트에 가려 노출 불가였던 걸 *직접 사용 중* 발견·제거.
- **PWA 범위 의도적 한정** — 설치형 셸까지만(푸시·오프라인 제외)하고 그 이유를 기록. "왜 안 했는가"도 결정으로 남김.
- **설치 배너 생명주기 처리** — `beforeinstallprompt`를 잡아 커스텀 설치 UI를 띄우되, iOS 사파리는 미지원이라 "공유 → 홈 화면에 추가" 수동 안내로 분기. 설치·닫기 시 `localStorage`에 기억해 재노출을 억제하고, 이미 설치(standalone)면 숨긴다 — *설치한 사용자를 다시 귀찮게 하지 않는* 실전 엣지케이스까지 챙김.
- **경로 따라가기(포그라운드 실시간 추적)** — `watchPosition`으로 현재 위치를 지도에 실시간 표시 + 이동 거리 누적(GPS 노이즈 <8m 무시), Wake Lock으로 화면 유지(미지원 시 무시). 웹의 백그라운드 한계를 인정하고 *포그라운드 전용*으로 범위를 못 박음.
- **무작위 코스 생성의 견고성** — 매번 다른 방향을 뽑되 두 코스를 90~270° 벌리고, 강·막다른 길 등 *경로 생성 실패 방향은 새 방향으로 재시도*. 외부 라우팅(Google/OSRM) 실패엔 폴백 제공자까지 둠.

## ⑤ 결과 & 남은 한계 · 다음
- **결과** — 기획→설계→구현→배포→재설계까지 1인 end-to-end. 측정에서 멈추던 제품에 '완수' 고리를 더해 **완수율을 측정 가능한 제품으로** 전환.
- **남은 한계** — 막 출시해 실사용 데이터가 없음 → **완수율 검증이 다음 과제.** 검증 전엔 GPS·네이티브에 베팅하지 않기로 명시.
- **다음 계획** — 완수율이 검증되면 네이티브(백그라운드 GPS·푸시)가 자연스러운 다음 플랫폼. 익명 루프를 비용 한도 내 어디까지 열지 결정.
- **추정 정확도의 다음 단계** — ① 칼로리는 사후 GPS 측정으로 *실제값* 계산, ② 경로는 **경사(고저차) 반영**(Google Elevation API) — 같은 거리라도 언덕은 더 태우므로 "평지/언덕" 차별화 + 경사 보정 칼로리가 가능. 둘 다 API 비용 때문에 검증 후로 보류.
- **산출물** — 위 결정들을 **ADR(결정 기록)로 문서화**해 판단 근거를 추적 가능하게 함. (`docs/adr/` 참고)
