# Eat & Run 포트폴리오 정리

## 1) 프로젝트 한 줄 소개
- 음식 사진(또는 텍스트) 분석 결과를 기반으로, 사용자가 바로 실행 가능한 러닝 계획과 경로를 추천하는 실행형 헬스 루틴 서비스.

## 2) 해결하려던 문제
- 기존 식단/운동 앱은 기록에 치우쳐 있어 실제 실행 전환이 약함.
- 사용자는 "오늘 뭘 먹었고, 얼마나 운동해야 하는지"를 빠르게 알고 싶어함.
- 입력(음식)에서 실행(러닝)까지 단계를 짧게 만들어 이탈을 줄이는 것이 핵심 목표.

## 3) 내가 맡은 역할
- 프론트엔드/백엔드 기능 구현 및 배포 파이프라인 구축.
- 분석 API 연동, 사용자 흐름 설계(분석 -> 활동 설정 -> 경로 추천 -> 기록).
- 보안/운영 관점 개선(호출 제한, 키 분리, 환경변수 관리).

## 4) 사용 기술 스택
- Frontend: Next.js(App Router), React, TypeScript, Tailwind
- Backend: Express, OpenAI API, Multer
- Data: Supabase
- Deploy: Vercel(프론트), Render(백엔드)

## 5) 핵심 기능
- 사진 기반 음식 칼로리 분석
- 텍스트 기반 음식 칼로리 분석(사진 외 입력 수단 추가)
- 사용자 목표 칼로리 기반 운동 방식/시간 계산
- 지도 기반 경로 추천(Google/OSRM fallback)
- 기록 저장/조회/삭제(개인 식별 쿠키 기반)

## 6) 기술적으로 강조할 포인트
- 입력 방식 확장: 단일 이미지 입력에서 사진/텍스트 탭 구조로 확장.
- 모듈화 리팩터링: 분석 화면을 탭/폼/결과 카드 단위로 분리해 유지보수성 개선.
- 프록시 아키텍처: Next API에서 백엔드 호출을 중계해 클라이언트와 백엔드 결합도 완화.
- 장애 완충: 외부 라우팅 실패 시 OSRM fallback 적용.

## 7) 보안/운영 개선 포인트
- API Key 관리:
  - 브라우저 공개 키(`NEXT_PUBLIC_GOOGLE_MAPS_API_KEY`)와 서버 키(`GOOGLE_MAPS_API_KEY`) 분리
  - 키 제한(Referrer/API restriction) 적용
- 호출 제한(Rate Limiting):
  - 사용자 기준 + IP 기준 이중 제한 적용
  - 엔드포인트별 상이한 제한값 적용
  - 429 응답 및 프론트 안내 메시지 처리
- 환경변수 분리 및 배포별 설정 문서화

## 8) 배포 구조 설명
- 프론트: Vercel
- 백엔드: Render
- 프론트는 `/api/v1/*` 경유로 백엔드와 통신
- 배포 시 프론트/백엔드 동시 반영이 필요한 변경(예: 새 분석 엔드포인트)을 분리 체크리스트로 관리

## 9) 트러블슈팅 사례(면접용)
- 증상: 로컬에서 텍스트 분석 실패
- 원인: 신규 백엔드 엔드포인트 추가 후 서버 재시작 누락(구버전 프로세스 실행)
- 해결: 백엔드 재시작 및 배포 환경(Vercel/Render) 동시 재배포
- 배운 점: API 계약 변경 시 "코드 반영 + 프로세스 재기동 + 양쪽 배포"를 한 세트로 관리해야 함

## 10) 이 프로젝트로 말할 수 있는 성장 포인트
- 기능 구현뿐 아니라 배포/보안/운영 이슈까지 통합적으로 다루는 경험.
- 단일 기능 개발에서 끝내지 않고, 실제 사용자 시나리오(실행 전환) 중심으로 개선.
- 문제가 생겼을 때 원인 분리(환경변수, 배포 버전, 엔드포인트 계약)를 체계적으로 수행.

## 11) 이력서/포트폴리오에 바로 쓸 문장 예시
- "Next.js + Express 기반 건강 루틴 서비스에서 음식 분석 결과를 러닝 계획으로 연결하는 end-to-end 흐름을 구현했습니다."
- "사진 입력에 더해 텍스트 분석 입력을 추가해 사용자 입력 접근성을 확장했습니다."
- "사용자/IP 이중 레이트리밋을 도입해 다계정 우회성 트래픽에 대한 방어력을 높였습니다."
- "Vercel-Render 분리 배포 환경에서 환경변수/키 정책/재배포 절차를 문서화해 운영 안정성을 개선했습니다."

## 12) 다음 업데이트 시 이 문서에 추가할 항목
- 실제 사용자 지표(분석 성공률, 분석 응답시간, 경로 추천 성공률)
- 개선 전/후 비교 수치(이탈률, 요청 실패율)
- 데모 링크, 시연 영상, 대표 화면 캡처

## 13) 최근 업데이트 요약 (2026-04)

### 인증/접근 제어
- 첫 체험 1회 이후 로그인 필수 정책 적용
  - `eat_run_trial_used` 쿠키 기반으로 비로그인 재접근 차단
  - 로그인 사용자는 전체 기능 접근 가능
- 로그인 방식 확장
  - 이메일 매직링크 + Google OAuth 지원
  - Kakao OAuth는 비즈 인증 완료 전까지 UI 임시 비노출 처리
- 페이지/서버 이중 가드 적용
  - `/analyze`, `/history` 페이지 진입 시 클라이언트 리다이렉트 가드
  - 주요 API에서 공통 권한 판별 로직(`resolveAccessContext`) 사용

### 데이터 일관성
- 게스트 기록 자동 이관
  - 로그인 직후 게스트 기록(`eat_run_uid`)을 로그인 계정으로 자동 마이그레이션
  - 이관 후 쿠키 식별자도 로그인 계정 기준으로 정렬

### 안정성/UX 개선
- 경로 추천 API 무한 재요청 방지
  - 401/실패 상황에서 자동 호출 루프 차단
  - 수동 재시도(`경로 다시 시도`)는 유지
- 기록 삭제 안전장치 추가
  - 개별 삭제/전체 삭제 모두 확인 모달 후 실행
  - 삭제 중 중복 클릭 방지
- 모바일 가독성 개선
  - `kcal` 단위 줄바꿈 방지 및 상단 요약 블록 반응형 정렬 개선

### 운영 관점 포인트
- 권한 판단 로직 중앙화 + 기능별 집행 구조로 확장성/일관성 확보
- 클라이언트 가드 + 서버 가드 병행으로 우회 가능성 최소화
- 로그인 전환 시 사용자 데이터(게스트 기록) 손실 최소화로 전환 마찰 감소
